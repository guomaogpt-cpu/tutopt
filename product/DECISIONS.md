# Product Decisions — Tutopt

Ключевые продуктовые и технические решения. Формат: контекст → решение → последствия.

---

## PD-001: B2B only, оптовые объявления

**Контекст:** рынок Кыргызстана, магазины и HoReCa ищут поставщиков оптом.  
**Решение:** платформа только для оптовых B2B-сделок, не retail.  
**Последствия:** MOQ обязателен, заявки вместо «купить в 1 клик», акцент на компании-продавце.

---

## PD-002: Заявки вместо прямых покупок

**Контекст:** B2B сделки требуют переговоров по цене, объёму, доставке.  
**Решение:** buyer отправляет Lead (заявку), seller связывается вне платформы.  
**Последствия:** нет корзины, оплаты, escrow на MVP.

---

## PD-003: Модерация перед публикацией

**Контекст:** качество каталога критично для доверия B2B.  
**Решение:** новые объявления → `PENDING_MODERATION` → approve/reject модератором.  
**Последствия:** задержка публикации, нужна админка модерации.

---

## PD-004: Роли: BUYER, SELLER, MODERATOR, ADMIN

**Контекст:** разные пользователи — разные сценарии.  
**Решение:** фиксированные роли в `UserRole` enum. Один user = одна роль.  
**Последствия:** SELLER регистрируется с `company_name`, MODERATOR назначается ADMIN.

---

## PD-005: Session auth, не JWT в localStorage

**Контекст:** безопасность, SSR в Next.js.  
**Решение:** httpOnly cookie + Session в БД.  
**Последствия:** проще logout/revoke, сложнее mobile API без доработки.

---

## PD-006: Уведомления через DB + polling (не WebSocket)

**Контекст:** MVP, минимум инфраструктуры.  
**Решение:** Notification model + polling unread-count каждые 30 сек.  
**Последствия:** задержка до 30 сек, нет real-time; зато простой deploy.

---

## PD-007: Повторные заявки разрешены

**Контекст:** buyer может уточнить условия, отправить новую заявку.  
**Решение:** нет unique constraint на buyer_id + listing_id; каждый POST = новый Lead + Notification.  
**Последствия:** seller видит несколько заявок от одного buyer.

---

## PD-008: Return URL после login

**Контекст:** гость терял контекст при входе с карточки объявления.  
**Решение:** `/login?next=<internal-path>`, валидация только путей начинающихся с `/`.  
**Последствия:** seamless return после auth для leads, favorites, protected pages.

---

## PD-009: Role-aware CTA «Подать объявление»

**Контекст:** BUYER/MODERATOR видели CTA, но получали denial.  
**Решение:**
- SELLER/ADMIN → `/listings/new`
- Guest → `/login?next=/listings/new`
- BUYER → `/listings/new` с понятным сообщением
- MODERATOR/ADMIN → CTA скрыт в header

---

## PD-010: Legacy routes redirect

**Контекст:** `/catalog`, `/categories` вели на несуществующие URL.  
**Решение:**
- `/catalog` → redirect `/listings`
- `/categories` → реальные категории из БД
- `/sellers` → placeholder «скоро»

---

## PD-011: Контакты продавца только для авторизованных

**Контекст:** стимул регистрации, защита от спама.  
**Решение:** телефон/email на профиле продавца и в карточке — после login.  
**Последствия:** гость видит CTA «Войти».

---

## PD-012: Локальное хранение изображений (MVP)

**Контекст:** быстрый старт без облака.  
**Решение:** upload в `/public/uploads/listings/`.  
**Последствия:** не масштабируется; S3 — в roadmap v0.2.
